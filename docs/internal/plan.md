## TECH STACK DECISIONS

| Decision | Choice | Reason |
|---|---|---|
| Bundler | **esbuild** | Fastest possible, zero config, watch mode built-in |
| Language | **Vanilla JS (ES2022 modules)** | No framework weight, direct DOM, portable |
| CSS | **Vanilla CSS with custom properties** | Full control, no purge issues, no build step for styles |
| Routing | **Hash router** (`#/dashboard`, `#/shifts`) | Works offline, no server config needed |
| Database | **Dexie.js** (IndexedDB wrapper) | Raw IDB is brutal; Dexie is 5x less code, typed, reliable |
| Charts | **Chart.js** | Complete, offline, no CDN needed at runtime |
| Dates | **Day.js** | 2kb, handles all shift math, plugins for duration/locales |
| CSV | **Papa Parse** | Edge-case-safe, stream-safe for large imports |
| Search | **Fuse.js** | Fuzzy search across all text fields, zero config |
| Drag & Drop | **Sortable.js** | Widget reorder, platform ranking, templates |
| Celebrations | **canvas-confetti** | Badge unlocks, goal completions |
| Screenshots | **html2canvas** | Year-in-review shareable image export |
| QR Codes | **qrcode-generator** | Feature 157, tiny, offline |
| Service Worker | **Hand-written SW** (simple) | Workbox adds weight; a 60-line SW is safer and auditable |

**All libraries are vendored locally in `/src/libs/` — zero CDN dependency at runtime.**

## SERVICE WORKER ARCHITECTURE (READ THIS FIRST)

This is the most critical architectural decision. Get this wrong and data feels fragile.

### The Rule: SW touches ONLY static assets. NEVER data.

```
┌─────────────────────────────────────────────────┐
│                  USER'S BROWSER                  │
│                                                  │
│  ┌──────────────┐      ┌───────────────────┐    │
│  │ Service      │      │    IndexedDB       │    │
│  │ Worker       │      │    (Dexie.js)      │    │
│  │              │      │                   │    │
│  │ Caches ONLY: │      │ ALL app data:     │    │
│  │ - index.html │      │ - shifts          │    │
│  │ - bundle.js  │      │ - expenses        │    │
│  │ - style.css  │      │ - vehicles        │    │
│  │ - libs/*.js  │      │ - goals           │    │
│  │ - assets/*   │      │ - platforms       │    │
│  │              │      │ - badges, etc.    │    │
│  └──────┬───────┘      └────────┬──────────┘    │
│         │                       │               │
│    Cache-first               Always             │
│    strategy                  in-memory          │
│    for shell                 + on disk          │
│                                                 │
└─────────────────────────────────────────────────┘
```

### SW Caching Strategy:
- **Cache name**: `macadam-shell-v{VERSION}` — version bump = old cache wiped
- **Cache-first** for: `bundle.js`, `style.css`, `index.html`, all `/assets/`, all `/libs/`
- **Network-first with cache fallback** for: `manifest.json`
- **Never intercept**: Any IndexedDB operation (those never go through fetch anyway)
- **On activate**: Delete all old cache versions except current
- **On install**: Pre-cache the full app shell manifest (generated by esbuild build script)

### Why NOT Workbox:
Workbox auto-generates SW from your build but adds 30kb+ and hides what's happening.
A hand-written SW is 60 lines, fully auditable, and you know exactly what's cached.
The build script generates a `CACHE_FILES` array that the SW imports — best of both worlds.

## ESBUILD CONFIGURATION

`build.js` — runs as `node build.js` (dev) or `node build.js --prod`

```js
// Key esbuild settings:
// - entrypoint: src/main.js → dist/bundle.js
// - format: iife (no ES module issues with older PWA installs)
// - bundle: true (inlines everything including libs)
// - sourcemap: true in dev, false in prod
// - minify: false in dev, true in prod
// - watch: in dev mode only
// - After build: run generate-sw-manifest.js to update CACHE_FILES
// CSS: esbuild does NOT process CSS. Use a simple concat script.
// CSS concat order: reset → tokens → themes → components → layout → animations → views/*
```

**package.json scripts:**
```json
{
  "dev": "node build.js --dev",
  "build": "node build.js --prod",
  "preview": "npx serve dist"
}
```

### F2 — Service Worker & PWA Shell
**Agent completes this entire task in one session.**
**Dependencies:** F1

#### Subtasks:
1. `public/sw.js` — hand-written service worker:
   ```
   - Import sw-manifest.js to get CACHE_FILES array
   - CACHE_NAME = 'macadam-shell-v1' (version bumped by build script)
   - install event: cache all CACHE_FILES, skipWaiting()
   - activate event: delete all caches not matching current CACHE_NAME, clients.claim()
   - fetch event: 
       if (request.method !== 'GET') return fetch(request)  // never cache mutations
       if (url is IndexedDB operation) return fetch(request)  // never — IDB bypasses fetch anyway
       try cache first → if miss → network → store in cache → return
       if network fails → return cached version or offline fallback
   - Never cache: requests with query strings, cross-origin requests, non-GET
   ```
2. SW registration in `src/main.js`:
   - Register only if `'serviceWorker' in navigator`
   - On update detected: show MacadamNotify banner "App updated — reload for latest version" with reload button (Feature 293 prep)
   - Store registration in module-level variable for later use (Background Sync, Feature 241)
3. `public/manifest.json` — full PWA manifest:
   - name: "Macadam", short_name: "Macadam"
   - icons: 192, 512, maskable variants
   - theme_color: pulled from CSS token (match `--color-brand`)
   - background_color: matches app background
   - display: "standalone"
   - start_url: "./#/dashboard"
   - scope: "./"
   - shortcuts: "Log New Shift" → `#/shifts/new`, "This Week" → `#/analytics/week`, "Export Vault" → `#/reports/export` (Feature 250)
4. `beforeinstallprompt` intercept in `src/main.js`:
   - Capture event, store it
   - Do NOT show native prompt immediately
   - Expose `window.__macadam.triggerInstall()` for the settings page to call (Feature 242)
5. Detect standalone mode: `window.matchMedia('(display-mode: standalone)')` → set `document.documentElement.dataset.installed = 'true'` → CSS can react to this (Feature 243)
6. Create placeholder 192×192 and 512×512 SVG icons (macadam road/asphalt pattern branding)

**Features covered:** 239, 240, 241, 242, 243, 250

### Vault-active gate (onboarding vs “real” session)

There is **no server login**: the vault always has a local `users` row (id `1`). During `#/onboarding` that user may already exist with partial data, so **presence of a user row is not enough** to treat the app as fully set up.

**Rule:** Product and financial surface nudges must run only when **`user.onboardingComplete === true`**.

- Implement **`isUserVaultActive(user)`** / **`isVaultActive()`** in `src/core/vault-gate.js` (or equivalent single helper).
- **P8** — `checkAllNotifications()` (and `createNotification` as a safety net) return immediately when the vault is not active, so no daily summary, tax, backup, goal, or arbitrage cards fire during onboarding.
- **P13** — changelog, tips, review nudge, and wellbeing toasts are skipped until the vault is active; **`window.__macadam.debug`** may still attach so developer utilities work during onboarding if needed.

The hash router already forces `#/onboarding` when incomplete; this gate prevents **parallel** notification/polish modules from firing on the same boot path.
